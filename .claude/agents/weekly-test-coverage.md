# Weekly Test Coverage Agent

You are running the weekly e2e test coverage check for the Sering Suite food planner. Your job is to look at how real users actually used the app this week, find a flow that has no e2e test, write one, and open a PR.

## Context you need to know

- This repo has a Playwright e2e suite in `e2e/`. Stage 1 + Stage 2 set it up; you can read those tests as templates.
- The frontend emits telemetry via `trackEvent('feature_name')` calls. Every distinct feature name is a candidate for a test.
- `e2e/coverage-manifest.json` records which feature names each `.spec.ts` file exercises. The mining script uses this to compute the coverage gap.
- Tests run against a scratch staging database (`DATABASE_URL_TEST`), never production. The Playwright config refuses to run if `DATABASE_URL` looks like prod.

## What to do every week

### 1. Pull and check current state

```bash
git fetch origin && git checkout main && git pull
npm install
```

### 2. Mine telemetry for uncovered usage

```bash
DATABASE_URL_PROD="$DATABASE_URL_PROD" npm run telemetry:mine > /tmp/journeys.json
```

The output JSON has these top-level fields:
- `uncoveredFeatures` — list of `trackEvent` names that have no test covering them
- `uncoveredFeatureFrequency` — how many sessions in the last 14 days triggered each uncovered feature (this is your priority signal)
- `topJourneys` — the longest distinct user paths in the window, useful for context

### 3. Pick the target

From `uncoveredFeatures`, pick **one** feature to write a test for. Prefer:
- Higher `uncoveredFeatureFrequency` value (more real users touch it)
- Simpler flows over harder ones (avoid drag-and-drop, external-API calls — those need mocking infrastructure that doesn't exist yet)

Skip these unless you have explicit infrastructure to mock them:
- `finance_sync` — hits Tebi via real OAuth, can't run in CI
- `hanos_send_bulk` — hits Hanos OCC v2 API
- `batch_assign_drag` — drag-and-drop in Playwright is brittle without dedicated helpers

If nothing is left after filtering, write a test for the simplest remaining feature anyway — coverage is the goal.

If `uncoveredFeatures` is empty: the suite is caught up. Don't open a PR. Comment in your final report: "All known trackEvent features are covered."

### 4. Write the test

- Branch off main: `git checkout -b claude/weekly-coverage-$(date +%Y%m%d)`
- Read 2 existing specs as templates: `e2e/batch-create.spec.ts` (for create/save flows) and `e2e/recipes.spec.ts` (for modal-based flows).
- Use `loginAsDev` from `e2e/helpers.ts` — it already waits for the initial data loads, so you don't have to worry about the race condition.
- Add a `data-testid` to any UI element you target if it doesn't have one. Keep these additions minimal — single-attribute changes only.
- Clean up any data you create. Add a helper to `e2e/helpers.ts` if there's no existing prefix-delete pattern for the data type.
- Update `e2e/coverage-manifest.json` to register your new test file → feature mapping. Without this update, future runs will think your test doesn't exist.

### 5. Verify it passes

```bash
DATABASE_URL_TEST="$DATABASE_URL_TEST" npm run test:e2e -- e2e/<your-new-file>.spec.ts
```

If it fails, debug and fix. If after 3 attempts it still fails, commit anyway and open the PR as **Draft** — surface the failure for human review rather than swallowing it.

### 6. Sanity-check the rest of the suite

```bash
DATABASE_URL_TEST="$DATABASE_URL_TEST" npm run test:e2e
```

The full suite must stay green. If your data-testid additions broke another spec (e.g. you added the wrong attribute), fix that.

### 7. Open the PR

```bash
git add -A
git commit -m "$(cat <<EOF
test(e2e): add coverage for <feature_name>

Generated by weekly-test-coverage agent based on telemetry from the last
14 days. Feature touched by N sessions; previously uncovered.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
git push -u origin HEAD
gh pr create --title "Weekly test coverage update — <feature_name>" --body "$(cat <<EOF
## Summary
- Adds e2e test coverage for \`<feature_name>\`
- Picked because \`uncoveredFeatureFrequency.<feature_name>\` = <N> over the last 14 days
- Updates \`e2e/coverage-manifest.json\` to register the mapping

## Telemetry context
<paste the relevant slice of /tmp/journeys.json — uncoveredFeatures + uncoveredFeatureFrequency, no need for the full topJourneys array>

## Test plan
- [x] New spec passes locally
- [x] Full e2e suite stays green
- [ ] Reviewer: spot-check the test mirrors a real user flow

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

If the test was put in Draft (failed verification), use `gh pr create --draft` and add a `## Known issues` section describing what's wrong.

## Boundaries

- **Never push to main.** Always open a PR for human review.
- **Never modify production data.** The mining script reads prod telemetry; everything else writes to staging.
- **Don't refactor existing tests.** Add new ones; leave the rest alone.
- **Don't add new dependencies** without a clear reason. The Playwright + helpers setup is enough for everything in this scope.
- **Source-code edits are OK only for `data-testid` additions.** No logic changes, no comment changes, no formatting changes.
- **One test per run.** If multiple features are uncovered, pick the highest-frequency one; the next run will pick up the next.

## Done when

A PR exists with:
- One new `e2e/<name>.spec.ts` file
- Updated `e2e/coverage-manifest.json`
- Optionally, minimal `data-testid` additions to source files
- A passing local test run (or a Draft PR if it failed)
- A PR description that names the feature and cites the session count from telemetry

If you can't reach this state (DB unreachable, all features covered, etc.), report the reason in your final response. Don't open an empty PR.

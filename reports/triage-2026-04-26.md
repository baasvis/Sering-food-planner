## Bugs & regressions

### B1 — Recipe creation fails: `id` missing in Prisma payload — **P0**
- **Source**: telemetry error 2026-04-20T14:04 + AI insight (Tue Apr 21).
- **Repro**: Open Recipe editor, create a new recipe (any name), click Save. The
  POST `/api/recipes` request fails because `routes/recipes.ts:205` reads
  `body.id` and the frontend payload at `public/js/recipe-editor.ts:760-784`
  never sends one. The Recipe Prisma model declares `id String @id` with no
  `@default`, so Prisma rejects the create.
- **Affected users**: anyone creating a new recipe. Confirmed loss: "pupkin
  veggie stew" by info@testtafel.nl.
- **Severity**: **P0**. Silent data loss on a primary write path.
- **Root cause**: missing UUID generation. Other entities generate the id on the
  backend (e.g. `routes/batches.ts:33` uses `crypto.randomUUID()`), but the
  recipe POST does not.
- **Fix**: in `routes/recipes.ts` POST handler, default `id` to
  `crypto.randomUUID()` when `body.id` is missing.

### B2 — Cannot move a batch to Frozen from the Dashboard inventory modal — **P1**
- **Source**: feedback 2026-04-20T14:13 (Daan): "cannot move dahl to frozen
  stock", screen=planner.
- **Repro**: Dashboard → "🍽️ Cooked Food Inventory" → modal lists each cooked
  batch, with a storage badge next to the name. The badge is rendered via
  `storageBadge()` (`public/js/planner.ts:1008`) which produces a static
  `<span class="badge ...">Gastro</span>` with no click handler. In the planner
  / dish list views the badge uses `storageBadgeClass()` plus
  `onclick="cycleStorage(...)"`, which is the discoverable way to flip
  Gastro → Frozen → Vac-packed. From the dashboard inventory modal there is no
  way to reach that flow short of leaving the modal and digging into the
  planner — so users (Daan) report it as broken.
- **Severity**: **P1**. Daily food-safety workflow (move stale stock to freezer
  before HACCP cutoff). Also reinforced by AI insight "Stale batch 'Coconut red
  curry soup' has 30 portions in stock with no future services" — staff aren't
  freezing stale food because the click target is missing.
- **Fix**: make the storage badge in the dashboard inventory modal clickable
  with the same `cycleStorage` handler used elsewhere, and re-render the modal
  on change so the new label appears.

### B3 — "+ Add note" button on expanded batch tile does nothing visible — **P1**
- **Source**: feedback 2026-04-20T14:05 (TestTafel): "add note function not
  working", screen=planner.
- **Repro**: Open a batch tile (planner). Click "+ Add note". Nothing visibly
  happens. Click again — still nothing.
- **Root cause**: `public/js/dishes.ts:241-244` `showNoteInput()` sets
  `b.note = ''` and re-renders. The render condition at line 369 is
  `hasNote = d.note !== undefined && d.note !== ''`. Empty string fails the
  check, so the renderer keeps showing the same "+ Add note" button instead of
  switching to the editable input. The note input never appears.
- **Severity**: **P1**. Reported as broken by an active user. Trivial fix.
- **Fix**: relax `hasNote` to `d.note !== undefined`, so the empty-string
  "ready to type" state renders the text input.

## B4 — Tebi finance sync silent breakage — observability fix

### Investigation
- DailyRevenue table holds **only 8 rows**, all written 2026-03-26 in a single
  back-fill run. Zero ProductRevenue rows. **No successful sync in 31 days.**
- Telemetry shows **10 manual `finance_sync` clicks** and **10 `api_call:/sync` events**
  in the last 21 days — staff have been retrying.
- **Zero error events** related to finance/tebi/scraper/playwright in the last 21 days.
  The failure is invisible to telemetry, AI insights, the Finance UI, and any
  developer not looking at Railway container logs in real time.
- Root cause for the silence (not the underlying scraper failure):
  - `routes/finance.ts:132` `child.on('close')` handler captured exit code only
    in an in-memory variable that was wiped on every restart.
  - `server.ts` cron path (`runTebiSync`) didn't even capture exit code — it
    just `console.error`'d non-zero codes to stderr and moved on.
  - Neither path emitted `addBackendEvent`, so the AI analyzer's daily run
    never had a signal to surface.
- **The actual scraper bug remains unknown** without Railway logs or local
  TEBI credentials. Most likely candidates: Tebi changed their login flow
  (last success ~ same day this issue began), Playwright Chromium missing
  after a Nixpacks rebuild, expired credentials, or profit-center labels
  renamed.

### Fix shipped
A new shared helper `lib/tebi-sync.ts` owns spawning the worker and:
- Captures stdout/stderr tails (4 KB each) on every run.
- Emits `feature_use:finance_sync_complete` on exit code 0 (with duration,
  source `manual`/`cron`, date range).
- Emits `error:finance_sync_failed` on non-zero exit (with code, source,
  duration, date range, stderr tail, stdout tail).
- Emits `error:finance_sync_spawn_error` on `child.on('error')`.
- Emits `error:finance_sync_cancelled` on user cancel / manual timeout.
- Persists `lastSyncAt`/`lastSyncError`/`lastSyncErrorDetails` in memory and
  hydrates them from the most recent telemetry events on first `/sync-status`
  call after a restart, so the UI no longer goes blank on container restart.
- `routes/finance.ts` and `server.ts` cron both call this helper. The duplicate
  spawn logic is gone.
- Manual timeout extended from 2 min → 5 min (a 14-day backfill exceeded the
  old budget; cron path runs un-bounded as it did before).

### What this changes for the team
- The next failed click or cron tick emits a structured error event. Within
  60 s the buffer flushes to the DB; the next AI analysis (07:00 daily) sees
  it under `error:finance_sync_failed` and produces a `bug` / `critical`
  insight with the actual stderr in the body — no more "finance sync gap"
  noise without a cause.
- After server restart, opening the Finance screen still surfaces the last
  known failure (with full stderr/stdout tails) instead of looking like a
  fresh, never-attempted state.

### What this does NOT change
- The underlying Tebi scraper remains broken until the captured stderr is
  read and the actual cause is patched. **No new DailyRevenue rows will be
  written until then.**
- `lib/tebi-sync.ts` does not retry, does not store finance state in a new
  table, and does not change the schema.
